A wireframe is a low-fidelity layout using boxes that resemble wires. The intent is to map out the site structure like flow—navigation, sections, and define priorities—without final colors or imagery. It lets us validate the UX quickly before further design and development commitment. This is just a building block for clarity and the developers understanding, and not a final draft.

A mockup is a medium‑ or high‑fidelity visual design that shows what the final website will actually look like. Unlike a wireframe, a mockup includes real colors, fonts, imagery, spacing, and layout behavior—it’s a full‑color preview of your site before any code is written. Mockups help you see how your brand, content, and features come together visually and make it easier to refine or request changes before development starts.

A hi‑fi mockup package in our process means you get the finished, pixel‑precise design plus clear documentation: written requirements, UI rules, component notes, and interaction guidance. This package becomes the single source of truth for both of us in the build phase—keeping expectations aligned and ensuring the implementation matches what you approved.

Content can arrive in phases, but earlier is better. A simple checklist:

  • Brand assets: logos (SVG/PNG), colors, typography, brand guidelines (if any).
  • Copy: headlines, body text, CTAs, product/service descriptions.
  • Images/video: high-resolution originals or preferred stock; we’ll optimize to WebP/AVIF.
  • Links & integrations: booking, forms, calendars, payment, or other tools to connect.
  • Legal: privacy policy, terms, and any disclosures.

Don’t worry if it’s not perfect—wireframes and the hi‑fi mockup package help reveal gaps early, and we can supply placeholder content where needed.

Acceptance criteria are simple, testable statements that define exactly what “done” means for each part of your project. They remove ambiguity and make sign‑off objective.

Examples:

  • “The ‘Contact’ button navigates to the contact page and the form submits successfully with validation errors displayed inline.”
  • “On screens under 640px, the navigation collapses into a keyboard‑accessible menu.”

We write criteria in plain language so anyone can check them. They protect scope, keep feedback focused, and give everyone a fair, shared definition of success.

QA stands for Quality Assurance—a systematic review to find and fix problems (bugs) before launch.

  • Functional: forms submit, buttons/links work, redirects and error states behave correctly.
  • Visual/Responsive: layouts hold on common devices and browsers; typography and spacing are consistent.
  • Accessibility basics: alt text, color contrast, focus order, and keyboard/screen‑reader checks.
  • Performance: page weight, image optimization, caching, and Core Web Vitals considerations.

Dedicated QA and bug‑fix time is included in every project so users experience a polished, reliable site from day one.

A repository ("repo") is the version‑controlled home for your code (e.g., GitHub). Our approach is straightforward: Catalyst retains ownership of the core codebase, and you always have permission to fork (copy) the full repository at any time—no restrictions. You can hand it to another developer, extend it internally, or redeploy elsewhere whenever you like.

This model keeps our portfolio rights intact while protecting your independence. You are never locked in, and we’ll provide clear handoff notes so your fork is easy to run and extend.

During the project, the repo lives in Catalyst’s GitHub for security and consistency. At any time you can fork a complete copy to your own account and continue development there. We include instructions for forking, environment variables, and deployment so you (or a new developer) can pick up smoothly.

Staging is a private, password‑protected environment used for reviews and QA. It mirrors production closely so we can test real data and integrations safely. Production is the public site your customers see. We deploy to staging first, address feedback, then promote the approved build to production.

Yes—production deployments are versioned so we can roll back quickly if needed. For dynamic sites we recommend scheduled database and asset backups via your host. We’ll outline the backup/restore workflow in your handoff notes.

Your domain is the address (example.com). Hosting is the server where your site files and APIs run. SSL encrypts traffic (the lock icon) to protect users and improve trust/SEO. We help connect your domain, provision SSL, and choose hosting that fits your stack and budget.

We build forms with validation, spam protection (honeypots/reCAPTCHA options), and clear success/error states. For reliable delivery, we recommend sending through your domain with authenticated DNS (SPF, DKIM, DMARC) or a transactional provider (e.g., Postmark, SendGrid). We’ll document what’s set up and how to monitor it.

A responsive site adapts fluidly to phones, tablets, and desktops without separate codebases. We design mobile‑first layouts, flexible grids, and media that scale—so content remains readable and interactive across breakpoints.

We target current versions of modern browsers: Chrome, Safari, Firefox, and Edge. On mobile we test iOS Safari and Android Chrome. Older or discontinued browsers (e.g., IE) aren’t supported by default.

If your audience relies on a specific platform or kiosk environment, tell us up front and we’ll include it in the test matrix.

SEO stands for Search Engine Optimization. It’s the process of improving your website so that it shows up higher in search engine results (Google, Bing) when people look for relevant terms. Higher rankings lead to more visibility, traffic, and potential customers.

Baseline SEO includes semantic HTML, descriptive titles/meta, Open Graph tags, sitemap/robots files, clean URLs, structured content patterns, and performance/accessibility fundamentals. We also ensure analytics is ready so you can measure results.

We build with a performance‑first mindset: semantic HTML, lean scripts, image optimization (WebP/AVIF where supported), responsive images, lazy‑loading, caching/CDN configuration, and Core Web Vitals awareness. Performance also feeds into SEO and accessibility.

Yes. We can set up Google Analytics or privacy‑friendly alternatives (Plausible, Umami). You own the accounts; we configure events/goals and a simple dashboard so you can track traffic and key actions.

We implement baseline privacy practices: minimizing third‑party scripts, honoring “Do Not Track” where possible, and loading analytics in a privacy‑aware way. If your business requires consent banners (e.g., GDPR/CCPA), we can integrate a compliant banner and cookie categories. We’ll help you connect it to your analytics so tracking respects user choices. (This is not legal advice; bring your counsel if you need specific language.)
